## Context

This is a greenfield project: there is no existing GUI codebase to integrate with. We are choosing a stack now and need to lock in decisions that affect every later contributor — backend selection, build system, dependency strategy, and platform lifecycle handling. The four targets fall into two natural families:

- **Desktop family (macOS / Windows / Linux):** Identical model — one process, one window, OS-provided event loop driven from `main()`. GLFW abstracts windowing and input across all three.
- **Mobile family (iOS):** Different model — Apple owns the app lifecycle (`UIApplicationMain`), there is no `main()` loop we control, rendering must be Metal (GL ES is deprecated), and windowing comes from `UIView`/`CAMetalLayer`. GLFW does not target iOS.

Pretending these are the same family (e.g., trying to force GLFW + GL on iOS) is a known dead-end. The design accepts the split and isolates it behind a thin platform-host layer.

Dear ImGui already ships backends for both stacks (`imgui_impl_glfw`, `imgui_impl_opengl3`, `imgui_impl_metal`, `imgui_impl_osx` and an iOS Metal example), so we are composing tested pieces rather than writing new backends.

## Goals / Non-Goals

**Goals:**
- One shared `app/` C++17 module that owns all UI code and is identical across platforms.
- Per-platform host (`platform/desktop/`, `platform/ios/`) thin enough that adding business logic never requires editing it.
- `cmake --preset <platform> && cmake --build --preset <platform>` produces a runnable artifact on every desktop target.
- iOS builds via an Xcode project generated by CMake; running on a device requires only a signing identity (no extra code changes).
- Pinned Dear ImGui revision — reproducible builds.
- A single, visually identical sample UI on first run, so "it works" is verifiable by eye on every platform.

**Non-Goals:**
- No HAL / virtual rendering interface (one desktop renderer, one iOS renderer — abstractions cost more than they save at this scope).
- No Android, no web/Emscripten (can be added later without re-architecting).
- No automated signing, notarization, TestFlight, or store submission.
- No custom widget library on top of ImGui.
- No high-DPI / multi-window / multi-viewport polish in this change (docking branch supports it; we wire the basics but don't productize).

## Decisions

### D1: Dear ImGui docking branch, pinned by git SHA
We use the `docking` branch rather than `master`. It includes the docking and (optional) multi-viewport features that are standard in modern ImGui apps; the upstream maintainer treats it as the de-facto default for application use.
- **Pinning:** A specific commit SHA, not a moving tag. FetchContent or a git submodule — see D2.
- **Alternatives considered:** `master` branch (less common in real apps, missing docking); Nuklear or other immediate-mode libs (much smaller community, fewer platform backends).

### D2: Vendor Dear ImGui via CMake `FetchContent` (preferred) with submodule as fallback
`FetchContent` keeps the source tree clean and locks the SHA in `CMakeLists.txt`. If we need to patch ImGui locally, we switch to a submodule. Either way, we compile ImGui ourselves as a static library — no system-installed ImGui.
- **Alternatives:** vcpkg / Conan — adds another tool to the contributor prerequisites for marginal benefit; rejected.

### D3: Desktop backend = GLFW + OpenGL 3.3 core profile
- GLFW is the standard cross-desktop windowing layer ImGui uses in its own examples.
- OpenGL 3.3 core is supported by every desktop GPU we care about and works identically on all three OSes. Avoids the Metal-only / D3D-only / Vulkan-only split.
- **Alternatives:**
  - **SDL2 + GL3:** Equivalent functionally. We pick GLFW because the ImGui example code and docs use it as the default; smaller surface area.
  - **Metal on macOS (native):** Lower latency, future-proof (Apple is deprecating GL). We keep this as an *optional* desktop-mac path documented in design but **out of scope for v1** — the cross-platform GL path is enough to ship.
  - **Vulkan everywhere:** Vastly more setup code (instance / device / swapchain / per-frame command buffers) for no user-visible benefit at this scope.

### D4: iOS backend = Metal + UIKit (no GLFW)
- iOS has no GLFW port and GL ES is deprecated.
- We host ImGui inside a `UIViewController` whose view is a `CAMetalLayer`. The ImGui Metal + iOS example in the ImGui repo is the reference implementation.
- The app lifecycle is owned by `UIApplicationMain` — our shared `app/` core exposes `init() / tick() / shutdown()` hooks that the iOS host calls from the view controller, and the desktop host calls from `main()`.
- **Alternatives:**
  - **GL ES on iOS:** Deprecated by Apple, will not pass App Store review for new apps long-term; rejected.
  - **MoltenVK (Vulkan→Metal):** Adds a dependency and complexity for no functional gain; rejected.

### D5: Shared app core exposes a lifecycle interface, not a main loop
The shared `app/` core does NOT own the main loop. It exposes:
```
app::init(RenderContext&);    // called once after backend is ready
app::frame(RenderContext&);   // called every frame, runs ImGui code
app::shutdown();              // called once before backend teardown
```
The desktop host drives this from a `while (!glfwWindowShouldClose) { ... }` loop. The iOS host drives `frame()` from a `CADisplayLink` callback. Neither host is visible to `app/` — the core compiles identically on all four platforms.
- **Alternatives:** Force `app/` to own the loop — impossible on iOS because Apple owns it. Inversion-of-control is the only design that works for both families.

### D6: CMake ≥ 3.24 with presets, single root `CMakeLists.txt`
- One top-level `CMakeLists.txt`, presets in `CMakePresets.json` for `macos`, `windows`, `linux`, `ios`.
- The `ios` preset sets `CMAKE_SYSTEM_NAME=iOS`, uses the Xcode generator, and disables the desktop host target. Desktop presets disable the iOS host target.
- **Alternatives:** Bazel (heavier toolchain commitment); separate per-platform projects (defeats the whole goal).

### D7: CI = build-only matrix for desktop, iOS documented as local-only
GitHub Actions runners cover macOS, Ubuntu, Windows for free. iOS builds need a signing identity and provisioning profile we don't want to commit; we document the local Xcode build steps and leave automated iOS CI for a future change.
- **Alternatives:** Set up unsigned iOS simulator builds in CI — possible, deferred to keep scope tight.

## Risks / Trade-offs

- **[Apple deprecates OpenGL further]** → Mitigation: D3 keeps an optional Metal-mac path in design.md; switching macOS to Metal later means writing one new desktop-mac host file, not re-architecting `app/`.
- **[ImGui docking branch occasionally has API churn]** → Mitigation: D1 pins a specific SHA. Upgrades become deliberate.
- **[Contributors on Linux can't test the iOS target]** → Mitigation: documented as expected; iOS is opt-in for contributors with a Mac + Xcode.
- **[CMake iOS toolchain is fiddly]** → Mitigation: Pin a known-good toolchain file (`ios-cmake` or CMake's built-in iOS support since 3.14), document required Xcode version, smoke-test on a real device once before declaring the spec met.
- **[GLFW + GL3 path has noticeable input latency on Windows compared to D3D11]** → Acceptable for a tool/utility shell; revisit only if we hit a real product requirement.
- **[Static-link bloat]** → Acceptable; binary is single-file deploy on desktop, app bundle on iOS.

## Migration Plan

Not applicable — greenfield project, no existing system to migrate from. Rollback = delete the new top-level `apps/imgui-shell/` directory and its CI jobs.
